## 1. 🎨 UI RELAYOUT — Streamlining Operations

### Problem: Excessive Navigation Depth

The current app has **29 page routes** with 4-5 click depth to accomplish basic tasks. The member workflow is especially fragmented:

```
/members → /members/:id → /members/:id/records → /members/:id/records/new
                        → /members/:id/timeline
                        → /members/:id/lab-records
                        → /members/:id/providers
                        → /members/:id/ai  (redirects to /conversations?memberId=)
```

### Key Findings

| Issue | Current State | Impact |
|-------|--------------|--------|
| **Member sub-routes** | 8 separate routes per member (`/records`, `/records/new`, `/timeline`, `/lab-records`, `/providers`, `/ai`, `/edit`, `/records/:id`) | Deep click-paths, context loss on navigation |
| **Dashboard fetches 7 APIs** on page load | `listMembers`, `getHousehold`, `listProviders`, `listReminders`, `listNotifications`, `listHouseholdRecords`, `listConversations` | Slow initial load, over-fetching |
| **Dashboard content is 851 lines** | `dashboard-content.tsx` — monolithic, hard to maintain | Poor separation of concerns |
| **Redundant sub-pages** | `member-records.tsx`, `member-providers.tsx`, `timeline.tsx`, `lab-records.tsx` are thin wrappers around content components | Unnecessary route entries |
| **AI Chat is disconnected** from member context | `/conversations` is standalone; member-specific AI is a redirect | Users lose their place |
| **Empty `ai/` and `chat/` component dirs** | Vestigial directories from cleanup | Dead code |

### Proposed: Unified Member Console + Streamlined Dashboard

**A. Merge member sub-routes into the tabbed `MemberTabs` component** (already partially done — `overview-tab`, `records-tab`, `timeline-tab`, `ai-tab` exist). The remaining sub-pages should be **deleted** and redirected to the tab:

| Route to Eliminate | Redirect To |
|---------------------|-------------|
| `/members/:id/records` | `/members/:id?tab=records` |
| `/members/:id/timeline` | `/members/:id?tab=timeline` |
| `/members/:id/lab-records` | `/members/:id?tab=records` (with filter) |
| `/members/:id/providers` | `/members/:id?tab=overview` (providers section) |
| `/members/:id/ai` | Already redirects — keep |

**B. Dashboard: Replace 7 API calls with single aggregated endpoint** (similar to `member_detail_service.py` pattern). The dashboard page currently:

- Fetches 7 separate endpoints
- Then fetches health scores individually per member (N+1)
- Then fetches member dashboards individually as fallback

Create a single `/dashboard/summary` endpoint that returns everything in one call.

**C. Eliminate 5-6 page components** that are thin wrappers:

- `member-records.tsx` → just renders `<RecordsContent>` with member context
- `member-providers.tsx` → just renders provider list
- `timeline.tsx` → just renders timeline component
- `lab-records.tsx` → filtered records view

These should all be tabs/sections within the Member Console.

## 7. ⚡ Performance Optimization Roadmap

### Frontend: Code-Splitting and React Rendering

- **Component Decoupling**:
  - Extract `HealthScoreRing` into `src/components/ui/health-score-ring.tsx` ✅ (already done)
  - Extract `FamilySummaryBar` into `src/components/members/family-summary.tsx` ✅ (already done)
  - Extract `MemberCard` and `MemberRow` into separate components under `src/components/members/` ✅ (already done)
- **Memoization (`React.memo`, `useCallback`, `useMemo`)**:
  - Memoize card render loops. Cards will only re-render if their exact biometrics, alerts, or score changes.
  - Wrap search-filtering inside `useMemo` so string matching is only performed when the raw list or search query actually mutates.
- **SWR Caching & Prefetching**:
  - Configure unified SWR hooks with `dedupingInterval: 15_000` to avoid dual-firing list requests on dashboard load.
  - Implement hover-based prefetching for member profiles: hovering over a card triggers SWR to fetch member detail in the background, rendering the console instantly upon click.

### Backend: Service Decoupling and Streamlining

- **AI Service Restructuring** (partially complete):

  ```
  backend/app/services/ai/
  ├── __init__.py
  ├── providers/              # Unified wrappers for LLM backends
  │   ├── __init__.py
  │   ├── base.py             # BaseProvider interface
  │   ├── gemini.py           # Google Gemini Adapter
  │   ├── openai.py           # OpenAI GPT Adapter
  │   ├── groq.py             # Groq LLaMA Adapter
  │   └── ollama.py           # Local Ollama client
  ├── document_extractor.py   # OCR parsing and structured data validation (Pydantic)
  ├── chat_assistant.py       # Chat context managers, prompt loading, & SSE streams
  └── insight_generator.py    # Health score audits, biometrics, & alerts engine
  ```

- **Concurrency & Event Loop Optimization**:
  - Replace any blocking JSON serializations or dictionary copies with async threads.
  - Audit `asyncio.gather` usages to ensure concurrent AI failover calls do not share mutable state.
